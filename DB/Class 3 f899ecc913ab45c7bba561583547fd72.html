DBMS Architecture:

What is a DB architecture? Database architecture refers to the design, structure, and organization of a database system. It defines how data is stored, accessed, managed, and secured in a database.

The database architecture helps to design, develop, implement and maintain a database management system.

Creating a DB architecture requires a very good and deep understanding of individual components of a database.

The DBMS interfaces is a way for the DBMS to communicate with users and applications.

Here’s a breakdown of some components:

  • Connection Manager: Allowing, rejecting and managing connections.
  • Security Manager: Managing the privileges among users and admins, the privileges allow level of control.
  • DDL compiler: Compiles the data definitions in DDL (Data Definition Language) into a machine readable form. After successful compilation (no issues), it registers the data definitions in the catalog.
  • Query processors allows the execution of database queries such as retrieval, insertion, update or removal of data. (SQL queries..). It is responsible for translating high-level database queries into low-level operations that the database engine can execute.
    • The query processor can be broken down into these components.
      1. DML compiler
      1. Query parser
      1. Query rewriter
      1. Query optimizer
      1. Query executer

  • Storage Manager: The storage manager controls physical file access and manages the correct and efficient storage of data.
    • It contains:
      1. Transaction Manager: supervises execution of database transactions. A database transaction is a sequence of read/write operations considered to be an atomic unit.
        • A database transaction is a sequence of operations performed as a single logical unit of work within a DBMS. These operations can include creating, updating, or retrieving data. The key aspect of a transaction is that it ensures the database remains in a consistent state, even in the event of errors or failures.
        • Transaction manager creates a schedule with interleaved read/write operations.
        • transaction manager guarantees ACID properties
          • ACID properties:
          • A = Atomicity - The entire transaction takes place at one or doesnt happen at all.
          • C = Consistency - The database must be consistent before and after the transaction.
          • I = Isolation - Multiple Transactions occur independently without interference.
          • D = Durability - The changes of a successful transaction occurs even if the system failure occurs

Tier Architecture

Tiers correspond to the level of machines or number of machines that the DB resides on.

Single level Tier means that the database is only stored on a single device and application.

Two-tier Client-Server Architecture

  • Clients use an interface to connect to a single database server. The application server and database server are the same. Before the application server was stored on the user’s end.

Three-tier Client-Server Architecture.

  • The clients connect to an application server which contains the application logic and processes, this application server then connects to a DBMS server for the data processing, which is then sent to the client. Single application on the server.

N-Tier Architecture

  • Another server is added in between the client and the application server, called the web server, which handles all web related tasks, related to networking and managing connections. This can when connect to the application server for processing the data.

Data Independence

  • A major object for three level architecture is to provide data independence which means that upper levels are unaffected by the lower levels.

Data independence refers to the ability to modify the schema at one level of a database system without affecting the schema at the next higher level. It's an essential concept in database management systems (DBMS) to ensure that changes in the data structure don't require changes in the application or its logic.

There are two types of data independence:

  1. Logical Data Independence: Changes in the logical schema (e.g., adding new fields or tables) should not affect the external schema or user views.
  1. Physical Data Independence: Changes in the physical storage of data (e.g., changing file structures or indexes) should not affect the logical schema.

Data independence promotes flexibility, scalability, and ease of maintenance in database systems.

The purpose of the 3-level architectural model is to enable data independence.
