Normalization

Normalizaiton is a process for assingin attributes to entities such that

  • Reduces data redundancies
  • Helps elimintae data anomalies
  • Produces controlled redundancies to link tables

Normalization is done in stages
X NF = X normal form

1 NF, 2 NF, 3 NF, 4 NF etc.

Each stage must be completed before moving to the other.

Attribute types from multiple entity types should not be combined in a single relation

Avoid excessive amount of Null values in a relation.

Normalization forms:

1 NF (First-Normal Form) → Table format, no repeating groups and PK defined.

Data anomalies

data anomalies are data items that are abnormal according to the rest of the data.

Types of anomalies

  1. Insert anomalies
  1. Delete anomalies
  1. Alter anomalies

Normalizaiton process

The objective of normalization is to ensure that each table confirms to the concept of well-formed relations, that is, tables that have the following characteristics

  • Each table represents a single subject
  • no data item will be uncessarilt stored in more than one table (in short, tables have minimum controlled redundancy).
  • All non-prime attributes in a table are dependent on the primary ket- the entire primary key and nothing but the primary key.
  • Each table is void of insertion, update, or deletion.

Dependency between attributes

Types of dependencies

Partial

  • Dependency on part of composite primary key

Transitive

  • One nonprime attributes depends on another nonprime attributes
  • X → Y and Y→Z then X→Z (X is a prime attribute)

Stages

1-NF (First normal form):

  • The first normal form ( 1 NF ) states that every attribute type of a relation must be atomic (One and Only one) and single valued.
  • Steps for converting to 1NF
    1. Eliminate the repeating groups
    1. Identify the primary key
    1. Identify all dependencies
  • Things to note
    • Multi-valued attributes are separated into different records.

2-NF (Second normal form):

  • A relation R is in the second normal form (2 NF) is it satisfied 1NF and every non-prime attribute type A in R is fully functional dependent on the key of R OR there are no partial dependencies.
  • If relation not in 2nd
    • Decompose it and set up a new relation for each partial key ogether with its dependent attribtue types
    • Keep a relation with the original primary key and any attribute thypes that are fully functional dependent on it.

3-NF (Third normal form):

  • Create seprarte tables to eliminate transitive functional dependencies
  • In 3NF
  • Contains no transitive dependencies

Improving the design

  • Minimuize data entry errors
    • Changing strings to constant ints
  • Evaluate Naming conventions
    • Changing attribute names so its more understandable which table they belong to.
  • Refine attribute atomicity
    • Dividing attributes
  • Identify new attributes
    • Consider real-world scenarios and add attributes that amplify the design.
  • Go back and identify relations that were originally left out.
  • Identify new relationships
  • Refine primary keys as required for data granularity.
    • Granularity regers to the level of detail represented by the values stored in a table’s row. Data stored at its lowest level of granularity is said to be atomic data.
  • Always try to have as less values in your composite key as possible
  • ER Diagram should go hand in hand with Normalization process.
